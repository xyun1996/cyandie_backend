# ADR 0001: Use Modular Monolith First

## Status

Accepted

## Context

本项目目标是通用后端框架，包含账号、平台接入、聊天、排行榜、后台管理等多个模块。第一阶段最重要的是快速形成可运行的核心能力，并保持模块边界清晰。

如果一开始采用微服务，会提前引入服务发现、分布式事务、跨服务调试、部署和观测复杂度，拖慢 MVP。

## Decision

第一阶段采用模块化单体：

- 一个主要后端应用。
- 通过模块边界隔离业务。
- 使用 MySQL 和 Redis。
- 使用 gRPC/Protobuf 定义后续可拆分的服务契约。
- worker 可以作为独立进程，但共享代码和数据库。

只有当模块边界稳定、性能或团队协作确实需要时，才拆分服务。

## Consequences

优点：

- 开发速度快。
- 本地调试简单。
- 数据一致性更容易维护。
- 更适合 agent 按模块渐进实现。

代价：

- 单体进程需要注意模块边界。
- 高并发 TCP 长连接和后台任务后期可能需要拆分。
- 需要靠测试和文档防止模块耦合。
